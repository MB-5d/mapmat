# Figma Parity Map

Active working contract: [figma-code-first-handoff.md](../../docs/design-system/figma-code-first-handoff.md)

Use this parity map as the audit baseline. Use the handoff contract for day-to-day Figma naming, repo ownership, and design-change workflow.

Audit basis: code-only review of the current frontend on 2026-04-11. No Figma file metadata, design library inventory, or Code Connect mappings exist in this repo, so the "Figma item" column is inferred from current code naming and reusable UI patterns.

## Comments Pilot Update
- What changed: this parity map now incorporates the 2026-04-13 comments-pilot captures across owner, commenter, and viewer sessions, expands comments from one partial feature row into a role-aware screen family, and adds explicit coverage for canvas surface styling, contextual right-rail shells, live status / permission banners, and shared-map role modes.
- Why the old parity map was insufficient: the previous code-only audit treated comments as a single partial feature row, which hid distinct screen contracts surfaced in the captures: panel populated vs filtered empty, editable popover vs reply + mention, commenter shared access vs viewer read only, and the live-view / reconnecting banners that frame those states.
- Top 5 highest-priority normalization targets:
  1. Shared `CommentsPanel` + `CommentPopover` + `CommentThreadItem` + `CommentComposer` family with explicit owner/commenter/viewer variants.
  2. Shared `ContextPanel` / right-rail shell for comments and future canvas-adjacent side panels.
  3. Shared `StatusBanner` / permission-banner component for live reconnecting, live view, and read-only states.
  4. Role-aware `NodeCardActionBar` + `CommentBadge` primitives with owner/commenter/viewer variants.
  5. Shared `Menu / Dropdown Surface` extended to mention suggestions and collaborator/account menus.

Status legend:
- `Matched`: clear shared code component or shell with a strong Figma equivalent.
- `Partial`: reusable pattern exists, but naming, tokenization, or state coverage is inconsistent.
- `Missing in Figma`: code surface exists, but there is no repo evidence of a defined Figma counterpart and it should be modeled explicitly.
- `Missing in Code`: strong Figma-worthy primitive is repeated inline and has no shared code abstraction yet.

## Top Priorities
- Continue the semantic token cleanup before binding more surfaces to Figma. `App.css` now defines the legacy aliases `--color-bg`, `--color-bg-secondary`, `--color-text-muted`, and `--radius-sm`, but the longer-term goal is to reduce alias usage and move more surfaces directly onto `--ui-*`.
- Extract shared code primitives for tabs/segmented controls, badges/pills, option cards, and menus/dropdowns. Those patterns already repeat across the product.
- Create explicit Figma components for the highest-value custom surfaces: `ScanBar`, `CanvasToolbar`, `MinimapNavigator`, `FeedbackWidget`, `ReportDrawer`, and admin surfaces.
- Move global chrome and core workspace surfaces off direct hex/rgba styling and onto semantic tokens, especially `Topbar`, `CanvasMapHeader`, `NodeCard`, and `ShareModal`.
- Decide whether the landing page is intentionally a second design-system track or whether it should converge with the app token vocabulary.
- Add Figma naming ownership and Code Connect only after the shared primitive set is stable.
